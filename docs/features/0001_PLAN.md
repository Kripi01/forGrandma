# 0001 – Compagnon agentique patient : vulgarisation et accompagnement du rapport d’imagerie

## Contexte

Application patient pour le hackathon Care Pathway in Oncology : vulgarisation et explication pédagogique du rapport d’imagerie, avec accompagnement du patient. Certains patients n’ont pas accès à une explication détaillée du rapport (surcharge des médecins). Objectif : un **compagnon agentique** — pipeline structuré (extraction, vulgarisation, validation, questions pour le médecin) + **chat conversationnel** ancré au rapport, sans jamais se substituer au médecin. Contrainte : **faisable en ~10h** ; **aucune persistance des données** (no data leaks, rapport uniquement en mémoire le temps de la requête).

## Fichiers et fonctions concernés

### À créer

- **Backend (nouveau)**  
  - Service ou API (FastAPI/Flask ou équivalent Node) exposant au minimum :  
    - `POST /api/report/understand` : entrée = texte du rapport ; sortie = JSON (extraction structurée, vulgarisation, résultat validation, questions pour le médecin).  
    - `POST /api/chat` : entrée = message utilisateur + contexte (extraction + vulgarisation + dernier échange si besoin) ; sortie = réponse de l’agent.  
  - Module(s) ou fonctions pour : orchestration du pipeline (4 étapes séquentielles), appels LLM par étape, et garde-fou (validation).  
  - Aucune base de données ; pas de stockage des rapports ou des conversations.

- **Frontend**  
  - **Composant « Résultat du pipeline »** : affichage des 4 blocs (extraction résumée, vulgarisation, indicateur « Explication vérifiée », questions à poser au médecin). Peut être une section dans la page principale ou un composant dédié (ex. `ReportExplanationPanel` ou équivalent).  
  - **Saisie du rapport** : soit zone de texte pour coller le rapport (si pas d’extraction PDF), soit couplage avec `PdfViewer` pour fournir le texte (extraction PDF côté client ou via backend). Dépend du choix d’implémentation (coller le texte minimise le scope à 10h).  
  - **État global ou partagé** : le résultat du pipeline (extraction + vulgarisation) doit être disponible pour le chat (contexte envoyé à chaque message).

### À modifier

- **`src/components/ChatPanel.tsx`**

- **Configuration**  
  - Variable d’environnement ou config pour l’URL de l’API backend (ex. `VITE_API_URL`).

## Algorithmes

### Pipeline agentique (séquentiel)

1. **Agent Extraction**  
   - Entrée : texte du rapport (anonymisé, pas de persistance).  
   - Rôle : extraire faits clés (localisation, taille, termes techniques, conclusion du rapport, niveau d’urgence si présent).  
   - Sortie : structure fixe (JSON) ou liste structurée (ex. `ExtractedFacts`).

2. **Agent Vulgarisation**  
   - Entrée : sortie Extraction.  
   - Rôle : produire 2–3 phrases par bloc (« Ce que montrent les images », « Ce que le médecin en conclut », « Ce que vous pouvez faire »). Règles : pas de diagnostic ajouté, pas de pronostic, toujours « à confirmer avec votre médecin ».

3. **Agent Validation (guardrail)**  
   - Entrée : texte vulgarisé.  
   - Rôle : vérifier l’absence de diagnostic inventé, de pronostic, de recommandation thérapeutique ; vérifier la présence de « parlez-en à votre médecin » (ou équivalent).  
   - Sortie : OK ou « à revoir ». Si à revoir : soit régénération (une fois), soit phrase de repli type « Demandez des précisions à votre médecin ».

4. **Agent Questions pour le médecin**  
   - Entrée : même contexte (extraction + vulgarisation).  
   - Rôle : générer 3–5 questions concrètes que le patient peut poser au prochain RDV (ex. « Faut-il refaire un scanner ? », « Que signifie “stabilité” pour mon cas ? »).  
   - Sortie : liste de questions (strings).

Orchestration : un seul flux séquentiel ; pas de boucle. Un seul LLM pour toutes les étapes est acceptable pour limiter coût et complexité.

### Chat ancré au rapport

- **Contexte** envoyé au LLM à chaque message :  
  - Résumé structuré (extraction),  
  - Texte vulgarisé validé,  
  - Derniers 3–5 échanges (historique court).  
- **Prompt système** : l’assistant aide à comprendre les termes et conclusions **du rapport** ; ne donne pas de diagnostic, pronostic ou traitement ; si la question dépasse le rapport, renvoyer vers le médecin ; s’appuyer uniquement sur les éléments fournis.  
- Pas de persistance : contexte et historique en mémoire (session) uniquement.

### Optionnel (si temps restant)

- **Glossaire** : à partir de l’extraction, lister les termes techniques du rapport et générer une définition en une ligne par terme (un appel LLM ou règle dérivée de l’extraction).  
- **« Explique-moi cette phrase »** : en cliquant sur un segment du rapport affiché, envoyer ce segment + contexte au même endpoint que le chat (ou dédié) et afficher l’explication dans un tooltip ou panneau.

## Phases recommandées

- **Phase 1 – Backend et types**  
  - Définir les types (ou schémas) pour : entrée rapport, extraction, vulgarisation, validation, questions, message chat.  
  - Implémenter le pipeline (4 étapes) et l’endpoint `POST /api/report/understand`.  
  - Implémenter `POST /api/chat` avec contexte extraction + vulgarisation + historique court.  
  - Aucune base de données ; tout en mémoire par requête / par session.

- **Phase 2A – UI pipeline et saisie rapport**  
  - Zone de saisie du rapport (coller le texte) et bouton « Comprendre mon rapport ».  
  - Appel à `POST /api/report/understand` et affichage des 4 blocs (extraction résumée, vulgarisation, indicateur validation, questions).  
  - Affichage optionnel d’une ligne du temps « Parcours » : Rapport reçu → Comprendre et préparer → Prochain rendez-vous.

- **Phase 2B – Chat et intégration**  
  - Brancher `ChatPanel` sur `POST /api/chat` en lui passant le contexte (résultat du pipeline).  
  - S’assurer que le chat n’est actif qu’une fois le pipeline exécuté (ou message clair « Chargez un rapport d’abord »).

Les phases 2A et 2B peuvent être menées en parallèle une fois la Phase 1 en place.

## Détails issus du brief

- **Focus patient** : surcharge des médecins → pas d’explication détaillée ; l’app comble ce manque par vulgarisation et accompagnement.  
- **IA agentique** : workflow en pipeline (extraction → vulgarisation → validation → questions) + agent conversationnel pour les questions de suivi.  
- **Pas de remplacement du médecin** : ton « compagnon » qui aide à comprendre et à préparer la consultation ; pas de diagnostic, pas de pronostic, pas de recommandation thérapeutique.  
- **Care pathway** : positionner l’outil comme une étape « Après réception du rapport, avant / pendant la prochaine consultation ».  
- **Pas de fuite de données** : aucun stockage des rapports ; suppression des copies exigée (règles du hackathon) ; tout en mémoire volatile.
